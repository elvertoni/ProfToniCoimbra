Dados que sobrevivem ao F5

O problema da memória curta

Pense em um site de compras. Você passa 20 minutos escolhendo produtos, adiciona cinco itens ao carrinho, vai até a página de pagamento... e, por acidente, aperta F5. Tudo desaparece.

Isso acontece porque, por padrão, as páginas web nascem com amnésia total. Cada vez que você entra, é como se fosse a primeira vez. Nenhuma informação da visita anterior sobrevive. Esse comportamento não é um defeito — é uma característica fundamental da web: o protocolo HTTP é stateless (sem estado).

✓

Conceito-chave

Persistência de dados é a capacidade de um sistema manter informações entre diferentes execuções, recarregamentos de página ou reinicializações. No front-end, a ferramenta nativa para isso é o LocalStorage.

i

Curiosidade

Em 1994, a Netscape criou os cookies — o primeiro mecanismo de persistência no navegador — para que um site de compras pudesse lembrar o que o usuário colocou no carrinho. A ideia deu tão certo que 30 anos depois ainda usamos cookies (e agora LocalStorage e IndexedDB) para a mesma finalidade.

!

Atenção

O LocalStorage não substitui um banco de dados real. Ele armazena dados apenas no navegador do usuário. Se o usuário trocar de computador, limpar os dados do navegador ou usar uma aba anônima, os dados somem. Para persistência real entre dispositivos, você precisa de um backend.

Nesta aula, vamos resolver o problema da "memória curta" usando as três tecnologias que você já conhece — HTML, CSS e JavaScript — mais uma API nativa do navegador que você vai dominar: o LocalStorage.

Formulários: a porta de entrada dos dados

Antes de salvar qualquer coisa, você precisa capturar os dados. E a ferramenta universal para isso na web é o formulário HTML.

Anatomia de um formulário

Um formulário é composto por três elementos fundamentais:

<form id="formAluno">
  <label for="nome">Nome completo:</label>
  <input type="text" id="nome" placeholder="Ex: João da Silva" required>

  <label for="turma">Turma:</label>
  <input type="text" id="turma" placeholder="Ex: 3DS" required>

  <label for="email">E-mail:</label>
  <input type="email" id="email" placeholder="Ex: joao@email.com" required>

  <button type="submit">Cadastrar</button>
</form>

O que cada parte faz:

  • <form> — agrupa todos os campos e define o comportamento de envio
  • <label> — rótulo acessível; o atributo for vincula ao id do campo
  • <input> — campo de entrada; type="text" para texto, type="email" para e-mail
  • required — atributo HTML5 que exige preenchimento antes do envio
  • <button type="submit"> — dispara o evento de envio do formulário
💡

Por que usar label + for?

Além de acessibilidade (leitores de tela), clicar no <label> automaticamente foca o campo vinculado. Isso melhora a experiência em dispositivos móveis, onde acertar um input pequeno com o dedo é difícil.

O evento submit — e por que pará-lo

Quando o usuário clica no botão Cadastrar (ou pressiona Enter), o formulário dispara o evento submit. Por padrão, o navegador:

  1. Coleta todos os dados dos campos
  2. Envia para o servidor indicado no atributo action do form
  3. Recarrega a página (comportamento padrão do HTTP)
!

Cilada comum

Se você esquecer o event.preventDefault() dentro do listener do submit, o formulário vai recarregar a página e você perde todos os dados não salvos. Esse é o erro nº 1 de quem está começando com JavaScript + formulários.

A solução é interceptar o submit com JavaScript e cancelar o comportamento padrão:

formAluno.addEventListener('submit', function(event) {
  event.preventDefault();   // 🛑 impede o recarregamento
  // ... seu código de processamento aqui
});

Roteiro do professor

"Pessoal, façam um teste mental: imaginem um site de cadastro que, ao clicar em 'Enviar', recarrega a página inteira e apaga tudo que vocês preencheram. Péssimo, né? Nosso trabalho como devs front-end é impedir exatamente isso. O preventDefault() é o superpoder que nos permite controlar o que acontece depois do clique, sem depender do comportamento padrão do navegador."

JavaScript no comando: DOM, eventos e dados

Com o formulário pronto e o evento submit sob controle, o próximo passo é ler os dados que o usuário digitou e organizá-los de forma estruturada.

Lendo valores do DOM

O DOM (Document Object Model) é a representação da página HTML como uma árvore de objetos JavaScript. Cada elemento vira um nó que pode ser lido ou modificado:

// Seleciona elementos pelo ID
const formAluno = document.getElementById('formAluno');

// Lê os valores digitados nos campos
const nome  = document.getElementById('nome').value.trim();
const turma = document.getElementById('turma').value.trim();
const email = document.getElementById('email').value.trim();

O método .trim() remove espaços extras no início e no fim do texto — uma validação simples mas essencial.

Organizando dados com objetos

Em vez de tratar cada campo como uma variável solta, usamos um objeto JavaScript para agrupar informações relacionadas:

const novoAluno = {
  nome:  "João da Silva",
  turma: "3DS",
  email: "joao@email.com"
};

// Acessando propriedades
console.log(novoAluno.nome);   // "João da Silva"
console.log(novoAluno.turma);  // "3DS"

Guardando múltiplos registros com arrays

Um aluno só é um objeto. Vários alunos formam uma lista — no JavaScript, um array:

const alunos = [];           // array vazio
alunos.push(novoAluno);      // adiciona no final
alunos.push(outroAluno);     // adiciona mais um

console.log(alunos.length);  // 2
console.log(alunos[0].nome); // "João da Silva"
✓

Modelo mental

Pense no array como uma gaveta de fichas. Cada ficha é um objeto (com nome, turma, e-mail). Você pode adicionar novas fichas (push), remover (splice), ou consultar (alunos[0]).

i

Conexão profissional

A estrutura "array de objetos" é o formato padrão de dados no front-end moderno. Frameworks como React, Vue e Angular trabalham exatamente assim: você recebe um array de objetos de uma API, renderiza na tela, e qualquer alteração nesse array atualiza a interface automaticamente. Entender objetos e arrays é pré-requisito para qualquer framework.

LocalStorage: o banco de dados do navegador

Agora que sabemos capturar e organizar dados, precisamos de um lugar para guardá-los. O LocalStorage resolve isso com uma API extremamente simples.

✓

Definição

O LocalStorage é um mecanismo de armazenamento chave-valor embutido em todos os navegadores modernos. Ele guarda strings de até ~5 MB por domínio. Os dados persistem após fechar a aba, reiniciar o navegador e até desligar o computador — só somem se o usuário limpar os dados do navegador manualmente.

As três operações fundamentais

// 1. SALVAR — guarda um valor associado a uma chave
localStorage.setItem("nome", "João");

// 2. LER — recupera o valor pela chave
const nome = localStorage.getItem("nome");  // "João"

// 3. REMOVER — apaga uma chave específica
localStorage.removeItem("nome");

A API é simples, mas tem uma limitação crucial: o LocalStorage só aceita strings. Se você tentar salvar um objeto diretamente:

const aluno = { nome: "João", turma: "3DS" };

// ❌ NÃO FAÇA ISSO
localStorage.setItem("aluno", aluno);
// Resultado: "[object Object]" ← perdeu os dados!
!

Por que "[object Object]"?

Quando o JavaScript tenta converter um objeto para string de forma implícita, ele chama .toString(), que retorna a string genérica "[object Object]". Nenhum dado real sobrevive a essa conversão. É por isso que precisamos do JSON.

Inspecionando o LocalStorage no navegador

Abra o DevTools (F12), vá na aba Application → Local Storage → selecione o domínio. Você verá todas as chaves e valores armazenados, podendo editar ou remover manualmente.

Chave Valor
alunos[{"nome":"João","turma":"3DS","email":"joao@email.com"}]
tema"dark"
💡

LocalStorage vs SessionStorage

Existe também o SessionStorage (sessionStorage.setItem()), que funciona igual mas apaga tudo quando a aba é fechada. Use LocalStorage para dados permanentes e SessionStorage para dados temporários de uma sessão.

JSON: a ponte entre JavaScript e armazenamento

Vimos que o LocalStorage só aceita strings, mas nossos dados estão em objetos e arrays. A solução é o JSON (JavaScript Object Notation) — um formato de texto para representar dados estruturados.

i

Contexto histórico

O JSON foi criado por Douglas Crockford em 2001 como um subconjunto da sintaxe de objetos do JavaScript. A ideia era simples: um formato que humanos conseguem ler, máquinas conseguem processar, e que já era nativo no JavaScript. Hoje, o JSON é o formato universal de troca de dados na web — toda API REST usa JSON.

As duas operações que você precisa dominar

const alunos = [
  { nome: "João",  turma: "3DS", email: "joao@email.com" },
  { nome: "Maria", turma: "3DS", email: "maria@email.com" }
];

// 1. CONVERTER JS → JSON (para salvar)
const jsonString = JSON.stringify(alunos);
console.log(jsonString);
// '[{"nome":"João","turma":"3DS","email":"joao@email.com"},{"nome":"Maria","turma":"3DS","email":"maria@email.com"}]'

// 2. CONVERTER JSON → JS (para ler/restaurar)
const dadosRestaurados = JSON.parse(jsonString);
console.log(dadosRestaurados[0].nome);  // "João"
✓

Regra de ouro

stringify quando for salvar (JS → texto) · parse quando for ler (texto → JS). Pense assim: stringify transforma em string, parse interpreta a string de volta.

Juntando tudo: localStorage + JSON

O padrão que você vai usar em todos os projetos:

// Função que carrega alunos do LocalStorage
// Se não existir nada, retorna array vazio
function carregarAlunos() {
  const dados = localStorage.getItem('alunos');
  return dados ? JSON.parse(dados) : [];
}

// Função que salva o array no LocalStorage
function salvarAlunos(alunos) {
  localStorage.setItem('alunos', JSON.stringify(alunos));
}

// Fluxo completo de um cadastro:
const alunos = carregarAlunos();     // 1. lê os existentes
alunos.push(novoAluno);              // 2. adiciona o novo
salvarAlunos(alunos);                // 3. salva tudo de volta
!

Erro clássico

Não fazer JSON.parse() ao ler. Se você usar localStorage.getItem() sem parse, recebe uma string JSON, não um array. Tentar dar .push() em uma string causa erro: "push is not a function".

Roteiro do professor

"Decorar stringify na hora de salvar e parse na hora de ler é a chave. Vou repetir isso várias vezes hoje. Se vocês saírem daqui sabendo só isso — stringify salva, parse lê — já valeu a aula inteira. O resto é detalhe."

📄 script.js — O coração do sistema. Leia os comentários com atenção:

// Seleciona os elementos da página
const formAluno = document.getElementById('formAluno');
const listaAlunos = document.getElementById('listaAlunos');

// ──────────────────────────────────────────
// FUNÇÕES DE PERSISTÊNCIA (LocalStorage)
// ──────────────────────────────────────────

// Lê os alunos salvos no LocalStorage
// Se não existir nada, retorna um array vazio []
function carregarAlunos() {
  const dados = localStorage.getItem('alunos');
  return dados ? JSON.parse(dados) : [];
}

// Salva o array de alunos no LocalStorage
// JSON.stringify converte o array para texto
function salvarAlunos(alunos) {
  localStorage.setItem('alunos', JSON.stringify(alunos));
}

// ──────────────────────────────────────────
// RENDERIZAÇÃO DA TABELA
// ──────────────────────────────────────────

// Exibe todos os alunos na tabela HTML
function renderizarTabela() {
  const alunos = carregarAlunos();

  // Limpa a tabela antes de redesenhar
  listaAlunos.innerHTML = '';

  // Para cada aluno, cria uma linha na tabela
  alunos.forEach(function(aluno, indice) {
    const linha = document.createElement('tr');
    linha.innerHTML = `
      <td>${indice + 1}</td>
      <td>${aluno.nome}</td>
      <td>${aluno.turma}</td>
      <td>${aluno.email}</td>
      <td>
        <button class="btn-remover" onclick="removerAluno(${indice})">
          Remover
        </button>
      </td>
    `;
    listaAlunos.appendChild(linha);
  });
}

// ──────────────────────────────────────────
// CRIAÇÃO DE REGISTROS
// ──────────────────────────────────────────

// Intercepta o envio do formulário
formAluno.addEventListener('submit', function(event) {
  // Impede o recarregamento da página
  event.preventDefault();

  // Lê os valores dos campos
  const nome  = document.getElementById('nome').value.trim();
  const turma = document.getElementById('turma').value.trim();
  const email = document.getElementById('email').value.trim();

  // Cria o objeto com os dados do aluno
  const novoAluno = { nome, turma, email };

  // Carrega, adiciona e salva
  const alunos = carregarAlunos();
  alunos.push(novoAluno);
  salvarAlunos(alunos);

  // Limpa o formulário e atualiza a tabela
  formAluno.reset();
  renderizarTabela();
});

// ──────────────────────────────────────────
// REMOÇÃO DE REGISTROS
// ──────────────────────────────────────────

// Remove um aluno pelo índice
function removerAluno(indice) {
  const alunos = carregarAlunos();
  alunos.splice(indice, 1);   // remove 1 item na posição 'indice'
  salvarAlunos(alunos);
  renderizarTabela();
}

// ──────────────────────────────────────────
// INICIALIZAÇÃO
// ──────────────────────────────────────────

// Ao carregar a página, exibe os alunos já salvos
renderizarTabela();

[Usuário] preenche nome, turma, e-mail
    ↓
[submit] evento disparado
    ↓
[preventDefault] bloqueia recarregamento
    ↓
[carregarAlunos] lê array existente do localStorage
    ↓
[push] adiciona novo aluno ao array
    ↓
[salvarAlunos] stringify + setItem no localStorage
    ↓
[renderizarTabela] tabela HTML atualizada

O que pode dar errado

1. Esquecer event.preventDefault() — o formulário recarrega a página e você perde os dados. 2. Não usar JSON.parse() ao ler — tenta dar .push() numa string. 3. Não chamar renderizarTabela() ao carregar a página — a tabela aparece vazia mesmo com dados salvos. 4. Caminho errado no <script src> — se não carregar o JS, nada funciona.

Além do LocalStorage: próximos passos

O LocalStorage é o ponto de partida, não o destino final. Vamos entender onde ele se encaixa no panorama completo de persistência de dados na web.

Hierarquia de persistência no front-end

Mecanismo Capacidade Persiste após F5? Persiste após fechar? Compartilha entre abas? Ideal para
Variável JS Ilimitada (memória) ❌ ❌ ❌ Dados temporários
SessionStorage ~5 MB ✅ ❌ ❌ Estado de uma sessão
LocalStorage ~5 MB ✅ ✅ ✅ Preferências, rascunhos, listas
IndexedDB ~50% do disco ✅ ✅ ✅ Dados estruturados pesados
Servidor + Banco Ilimitada ✅ ✅ ✅ (multi-dispositivo) Dados da aplicação real

Quando o LocalStorage NÃO é suficiente

  • Múltiplos usuários — cada um vê apenas seus próprios dados locais
  • Dados compartilhados — não dá para um usuário ver o cadastro do outro
  • Validação de segurança — qualquer um pode editar o LocalStorage pelo DevTools
  • Grandes volumes — o limite de 5 MB é pequeno para aplicações reais
  • Backup e sincronização — se o usuário trocar de máquina, perde tudo
✓

O caminho natural

Na próxima etapa, você vai aprender a substituir o LocalStorage por um servidor Node.js com banco de dados SQLite, mantendo exatamente a mesma estrutura de código no front-end. O fetch() substitui o localStorage.getItem(), e seus dados passam a ser acessíveis de qualquer dispositivo. A lógica de array, objetos e renderização que você aprendeu aqui continua idêntica.

i

Caso real

O Google Docs usa uma combinação de IndexedDB (para edição offline) + servidor (para sincronização). Você pode digitar um documento inteiro sem internet, e quando a conexão volta, tudo é enviado para o servidor automaticamente. O padrão "salvar local primeiro, sincronizar depois" é usado por WhatsApp Web, Trello, Notion e dezenas de apps modernos.
